# Artifact Workbench

The public template creates a private zip. The Flap Artifact Workbench owns production artifact creation.

Use `Artifact Workbench` as the platform name. It is the internal Flap workbench for source package upload, unpacking, validation, preview, review, artifact build, registry patch generation, and publish/disable/rollback decisions. Avoid using `intake` as the platform name; `intake` describes only the first submission handoff step.

## Source Package vs Runtime Artifact

There are two different outputs:

- `yarn vault:package <folder-name>` creates a source package for the Flap Artifact Workbench. It contains only the strict Vault package files: `Component.tsx`, `manifest.json`, `VaultABI.ts`, and `i18n.json`, plus the manifest schema, workbench metadata, script-generated package marker, and `qa/e2e-report.json`.
- Workbench validation should accept only source zips generated by `yarn vault:package <folder-name>`. The zip must include `flap-vault-package.json` with `kind: "flap-vault-ui-source-package"`, the supported format version, runtime package version and npm `gitHead` provenance, `artifactId`, folder name, required source file paths, matching SHA-256 hashes, and an `e2e` summary that matches `qa/e2e-report.json`.
- Manually assembled zips, zips missing `flap-vault-package.json`, zips missing `qa/e2e-report.json`, stale or failed E2E proofs, zips with duplicate entries, zips with mismatched hashes, or zips whose marker does not match the expected package kind/version must be rejected.
- The template includes `yarn vault:verify-package <zip>` as a local Workbench-side acceptance proxy. It checks the marker, package kind/version, exact file list, metadata consistency, E2E proof, and SHA-256 hashes before a package is handed off.
- The Flap Artifact Workbench builds the runtime artifact that `flap.sh` loads from Blob/R2/S3. That runtime artifact is browser-executable ESM JavaScript, not TSX source and not the source zip.

Auxiliary documents and sample data are not part of this template. Preview should use real wallet/runtime data and real addresses supplied through the runtime or preview URL.

The preview-side manifest panel is not a runtime artifact and is not packaged as Vault UI source. It exists only to show the current `manifest.json` while reviewing `/example` or `/{folder-name}` locally.

Runtime `component.mjs` should stay readable by default. Do not minify the MVP artifact unless Flap explicitly enables a release optimization step. If minification is enabled later, preserve source maps and the unminified source package in Workbench storage.

External endpoints, external resources, external frames, and fixed extra contract targets are discouraged. If a source package declares a special non-oracle endpoint in `manifest.json`, it must use a single absolute HTTPS URL string without username/password credentials or an array of those strings, and the Workbench must mark it for Flap review before production publish. If a source package declares a display-only chart frame, it may use only one `manifest.externalFrames[]` entry with `id`, `provider`, `src`, and `title`; providers are limited to TradingView, DexScreener, and CoinGecko Terminal/GeckoTerminal exact origins, and the `src` must be one complete static HTTPS URL with a fixed non-empty query string. If a source package must call a fixed contract that is not the runtime token, Vault, factory, or binding-scoped token/Vault reference, it must declare that target under `match.bindings[].externalContracts` with `address` and `label`. Oracle usage is detected by `vault:check` and provisioned by the Flap Artifact Workbench/runtime, not declared in the source manifest. Any external URL, frame URL, or fixed extra contract target used by code but not declared in the manifest is rejected. Direct `fetch(...)` must be static, absolute, HTTPS, credential-free, and covered by `manifest.endpoints`; host-relative, dynamic, HTTP, credentialed, aliased, destructured, or computed browser-global fetch targets are rejected because they cannot be reviewed precisely and can accidentally target private Flap app routes.

Endpoint, external frame, and external contract declarations are only review inputs. This public template does not define the approval SLA, approver, ticket queue, or production allowlist mechanism. Until Flap review explicitly approves the endpoint, frame, or contract target outside this template, agents must keep it in `openItems` with URL or address, purpose, request/response, frame/provider, or method shape, data sensitivity, fallback behavior, and the reason an SDK/on-chain/oracle/runtime-context path is insufficient.

When `vault:check` reports `manual-review/external-frame`, the output includes the full iframe `src` both in the warning issue and in `review.externalFrames[]`. Workbench should display those exact `src` values in the human review UI instead of asking reviewers to inspect `manifest.json` manually.

## Source Package Marker Schema

The `flap-vault-package.json` marker is the Workbench acceptance anchor. The Workbench verifier should treat this shape as required for package format version `4`:

```json
{
  "kind": "flap-vault-ui-source-package",
  "formatVersion": 4,
  "generatedBy": "yarn vault:package",
  "generator": "flap-vault-ui-template",
  "templateName": "flap-vault-ui-template",
  "templateVersion": "0.1.2",
  "runtimePackageName": "@flapsdk/vault-runtime",
  "runtimePackageVersion": "0.1.2",
  "runtimePackageGitHead": "40-char git sha from npm latest",
  "runtimeContractVersion": 1,
  "artifactId": "vaultui_<folder-name>_<26-char ULID>",
  "folderName": "<folder-name>",
  "sourcePackage": "src/vaults/<folder-name>",
  "packagedAt": "ISO-8601 timestamp",
  "check": {
    "passed": true,
    "summary": {
      "blocking": 0,
      "warning": 0,
      "info": 0
    }
  },
  "e2e": {
    "passed": true,
    "reportFile": "qa/e2e-report.json",
    "sourceSha256": "<source-sha256>",
    "manifestSha256": "<manifest-sha256>",
    "chainId": 97,
    "tokenAddress": "0x...",
    "tokenPolicy": "testnet",
    "viewportCount": 3,
    "phaseChecks": ["default", "internal-market", "dex-listed"],
    "layoutCheckSummary": {
      "horizontalOverflow": 0,
      "textOverflow": 0,
      "controlCovered": 0,
      "controlOverlap": 0,
      "riskStatus": 0,
      "wrongNetworkState": 0
    }
  },
  "requiredSourceFiles": [
    "src/vaults/<folder-name>/Component.tsx",
    "src/vaults/<folder-name>/manifest.json",
    "src/vaults/<folder-name>/VaultABI.ts",
    "src/vaults/<folder-name>/i18n.json"
  ],
  "fileSha256": {
    "src/vaults/<folder-name>/Component.tsx": "<sha256>",
    "src/vaults/<folder-name>/manifest.json": "<sha256>",
    "src/vaults/<folder-name>/VaultABI.ts": "<sha256>",
    "src/vaults/<folder-name>/i18n.json": "<sha256>",
    "schemas/manifest.schema.json": "<sha256>",
    "qa/e2e-report.json": "<sha256>"
  }
}
```

Validation requirements:

- `kind`, `formatVersion`, and `generatedBy` must match exactly.
- `templateVersion`, `runtimePackageVersion`, and `runtimePackageGitHead` must match the Workbench-supported npm latest `@flapsdk/vault-runtime` provenance.
- `folderName` must be valid lowercase kebab-case and must match every `src/vaults/<folder-name>/...` source entry.
- `check.passed` must be true and `check.summary.blocking` must be `0`; warning and info counts may be non-zero and remain review evidence.
- The zip must contain exactly the marker, `package-metadata.json`, `schemas/manifest.schema.json`, `qa/e2e-report.json`, and the four required Vault source files.
- `qa/e2e-report.json` must have `passed: true`, zero blocking issues, current hashes for the four Vault files and manifest schema, required PC / iPad / H5 viewports, required `default` / `internal-market` / `dex-listed` phase checks, and a valid BNB test token. Use chainId `97` first; chainId `56` is allowed only as mainnet fallback when no testnet token exists. The V1 report is deterministic Playwright DOM/layout/state proof, not AI image judgment.
- Duplicate zip entry names, path traversal entries, unsupported compression, mismatched central/local header names, unexpected files, or mismatched hashes are rejection reasons.
- `manifest.json` inside the zip must have the same `artifactId` as the marker.
- `package-metadata.json` must agree with marker kind, format version, folder name, artifact id, runtime provenance, and `e2e` summary.

For future write-UI checks, a tx hash or wallet trace in a developer-local report can prove only that a transaction exists and targets the expected token/Vault. It cannot strongly prove local UI origin. Strong write-UI origin assurance belongs in a platform-controlled Playwright + wallet runner, not in the V1 source package proof.

## Flow

```plain text
yarn vault:package my-vault
  -> private zip
  -> flap-vault-package.json marker
  -> Flap Artifact Workbench
  -> validation
  -> rejected JSON feedback or accepted review record
  -> endpoint/resource review when declared
  -> preview
  -> Flap build
  -> static artifact upload
  -> registry publish
  -> flap.sh runtime loader
```

## Workbench Feedback Loop

Workbench validation and review failures should be returned in the same agent-readable shape as local Vault CLI failures:

```json
{
  "ok": false,
  "code": "workbench/package-invalid",
  "error": "Human-readable failure.",
  "fixHint": "Regenerate the source package with yarn vault:package <folder-name> after fixing source files.",
  "agent": {
    "verdict": "fix-blocking",
    "nextActions": [
      {
        "ruleId": "package-verify/hash-mismatch",
        "severity": "blocking",
        "file": "src/vaults/my-vault/Component.tsx",
        "fixHint": "Do not edit the zip. Fix source, rerun vault:check/e2e/package/verify, then resubmit a new zip."
      }
    ]
  }
}
```

Rejected packages must not be patched in place. The resubmission loop is:

1. Fix the source package files or manifest in this template.
2. Run `yarn vault:check <folder-name>` and resolve all blocking issues.
3. Run `yarn vault:package <folder-name>`.
4. Run `yarn vault:verify-package dist/<folder-name>.zip`.
5. Submit the new zip and keep the new `sha256` in the handoff.

Endpoint review, oracle provisioning, registry binding, and runtime publish approval are Workbench/Flap decisions outside this template. Keep them in the Agent done report `openItems` until the decision is recorded.

After packaging succeeds, the script prints the source zip location:

```json
{
  "sourcePackagePath": "dist/my-vault.zip",
  "sourcePackageAbsolutePath": "/path/to/flap-vault-ui-template/dist/my-vault.zip",
  "packageMarkerFile": "flap-vault-package.json"
}
```

## Artifact Files

```plain text
component.mjs
manifest.json
i18n.json
metadata.json
review-report.json
```

`component.mjs` is generated by the Flap Artifact Workbench. Project developers should not hand-write or upload it directly.

`component.mjs` must export the built Vault component as the default export. The default export must be a React component compatible with the public `VaultComponentProps` contract from `@/src/sdk`: the host renders it inside the shared runtime provider and may also pass `{ sdk, context }` props for compatibility. It must not auto-mount itself, bootstrap a wallet/provider, mutate globals, or import private host files.

## Compatibility Versions

Source packages currently use package format version `4`. The Workbench must reject unsupported future versions unless it explicitly implements them, and this V1 policy rejects legacy format `3` packages for Workbench upload. Format version `4` requires npm `gitHead` provenance for `@flapsdk/vault-runtime` and a passing, source-hash-bound `qa/e2e-report.json`.

Versioning rules for `agent-contract.json`, `schemas/manifest.schema.json`, and package format changes live in `docs/versioning.md`.

## Runtime Artifact Key

The Vault folder name is the source folder and local preview route. `artifactId` is the stable unique source-package artifact identity and follows `vaultui_<folder-name>_<ULID>`. The Flap Artifact Workbench should use that `artifactId` to track the artifact family, while runtime build versions and storage paths remain Workbench-owned.

One runtime artifact can serve testnet, mainnet, future factories, fixed single-Vault deployments, or token-scoped deployments when the UI logic is the same. Deployment bindings map `chainId + factoryAddress`, no-factory `chainId + vaultAddress`, or no-factory `chainId + tokenAddress` targets to the shared artifact id. No-factory token scoping may carry multiple token CAs in `match.bindings[].tokenAddresses`. If it needs a fixed extra contract target, it should do so only as `match.bindings[].externalContracts`.

## Import Contract

Vault source may import:

- Public template/runtime aliases: `@/src/sdk`, `@/src/ui`.
- The single local relative file `./VaultABI`.
- Approved package imports already allowed by `vault:check`, such as `react`, `viem`, `decimal.js`, and `lucide-react`.

Vault source must not import private `flap.sh` app paths, host shell files, undeclared packages, `./helpers`, `../VaultABI`, nested components, local assets, any other local relative file, or dynamic imports. The Vault folder structure is fixed and must not be expanded.

The Flap Artifact Workbench must use the same alias contract when building `component.mjs`. The production runtime should provide the same public SDK/UI surface that local preview provides, so code that imports from `@/src/sdk` or `@/src/ui` works consistently in both places.

See [Runtime Module Contract](./runtime-module-contract.md) for the stricter build/runtime rule: Workbench and `flap.sh` should share one public runtime surface instead of separately bundling unrelated SDK/provider copies into every artifact.

## Storage Recommendation

MVP can use Vercel Blob public storage.

Rules:

- Never overwrite an artifact version.
- Disable through registry status, not by deleting files.
- Keep source zip or build backup.
- Validate hash before runtime load.
- Project developers must not have production Blob write access.
- Accept only Flap Artifact Workbench build output for runtime loading. Do not let project developers upload arbitrary `component.mjs` directly.
