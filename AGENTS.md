# Flap Vault UI Template Agent Guide

Use `yarn`.

This repository is a public template for controlled Flap Vault UI components. It must not include private `flap.sh` business code, secrets, or project-specific private configuration.

Core rules:

- Custom Vault UI is a controlled business component, not a free-form website container.
- For AI-agent-driven work, read `agent-contract.json` and `docs/ai-agent.md` before editing.
- Prefer `yarn vault:scaffold <folder-name> --chain 56 --factory 0x... --token 0x...` for factory-scoped UI, or `yarn vault:scaffold <folder-name> --chain 56 --vault 0x... --token 0x...` for single-Vault UI without a factory; it creates the strict four-file package and registers the preview folder name.
- Folder name is the source folder and preview route. Use 3-64 characters of lowercase kebab-case, for example `flap-nft-vault`; do not use spaces, underscores, uppercase letters, leading/trailing hyphens, or nested folders.
- `artifactId` is the unique artifact identity in `manifest.json`. `vault:scaffold` generates it by default as `vaultui_<folder-name>_<ULID>`; if you set it manually, keep its folder-name segment equal to the Vault folder name.
- Do not bind custom UI by type fields. Use `match.bindings[]` with `chainId` plus either a non-zero `factoryAddress` for factory-scoped UI, exactly one non-zero `vaultAddresses[]` entry for Vault-scoped UI without a factory, or one or more non-zero `tokenAddresses[]` entries for token-scoped UI without a factory. A zero factory address is invalid; omit `factoryAddress` for no-factory mode. Every manifest must include at least one valid binding-scoped `tokenAddresses[]` entry as the Workbench/E2E test token source; local `vault:e2e --token` overrides do not satisfy `vault:check`. Preview/runtime matching should prefer exact `chainId + factoryAddress`, `chainId + vaultAddress (+ optional tokenAddress)`, or `chainId + tokenAddress` matches and should not silently substitute an unrelated binding when a hint is ambiguous or mismatched. If the UI must call a fixed non-token/non-Vault/non-factory contract, declare it only as `match.bindings[].externalContracts[]` with `address` and `label`.
- `src/vaults/{folder-name}` is a strict package boundary. It may contain only `Component.tsx`, `manifest.json`, `VaultABI.ts`, and `i18n.json`; do not add any other files or folders.
- Prefer `@/src/sdk` and `@/src/ui`. Do not introduce any additional SDK package or SDK-like runtime wrapper inside a Vault package.
- Prefer CSS/HTML card shapes or `lucide-react` icons over ad hoc SVG. Search the official Lucide icon library first: `https://lucide.dev/icons/` (main site: `https://lucide.dev/`). FontAwesome is acceptable only in host repositories or runtimes that explicitly include and allow it; this template's Vault package allowlist does not include FontAwesome by default.
- Handwritten inline SVG JSX is allowed only for static pure graphic nodes such as `svg`, `g`, `defs`, `path`, `circle`, `rect`, `line`, `polyline`, `polygon`, `ellipse`, `linearGradient`, `radialGradient`, `stop`, `clipPath`, `mask`, `title`, and `desc`. It must not contain `script`, event attributes, `foreignObject`, `image`, `use`, external URLs, non-local `url(...)`, `style` `url(...)` / `@import`, `href` / `src` except static local fragments, spread attributes, or unsupported nodes.
- For UI style consistency, read `docs/ui-pattern-snippets.md` before implementing `Component.tsx`; use the snippets as sanitized organization patterns, not copied private code.
- Before implementing actions, decide whether buttons should be available in `internal-market`, `dex-listed`, `both`, or `read-only` stage. Use `context.host?.marketPhase` and `isActionAvailableForPhase(...)` for runtime gating. On supported preview chains, the template now reads the real Portal/helper/VaultPortal state from `chainId + tokenAddress`; the right-side preview panel exposes `真实` to restore the live host phase and `内盘` / `Listing` as explicit override tools for local self-test. `unknown` remains a runtime display value, not a primary tab. Production Flap host injects equivalent context. Show unavailable actions with a clear state instead of silently hiding them.
- Custom Vault UI in this template targets the tax-token path. Do not spend implementation effort on a separate non-tax custom-Vault branch. The live state that still matters is `context.host?.marketPhase` / `context.host?.isListed`, current contract risk status from `context.host?.vaultInfo?.riskLevel` or `context.host?.taxInfo?.vaultInfo?.riskLevel`, plus token metadata from the public SDK/host; do not introduce ad hoc props for that.
- Every onboarded Vault UI must read and visibly render the current contract risk status. Use `readTaxVaultHostContext(context.host)` and derive risk level from `host.vaultInfo?.riskLevel ?? host.taxInfo?.vaultInfo?.riskLevel`. The risk status must appear within the first three visible Vault-specific business rows/blocks and before any preview, hero, banner, showcase, media, chart, or large visual block; the safest placement is the first `CardHeader` badge row or first `Metric` / `DataRow` / `DetailTile` row. If the risk level is unavailable, render a prominent warning/danger message that risk-status integration is required; `vault:check` treats missing risk-status integration as blocking. Do not hardcode or unconditionally render `Low risk` / `低风险` labels, badges, summaries, or reassuring copy; a low-risk label may appear only when selected from the host-derived `riskLevel === 1` branch.
- For token images or names, use host-provided `context.tokenImageUrl`, `context.tokenName`, and `context.tokenSymbol`. The current template preview shell first asks the same-origin host-proxy for token presentation data, then falls back to on-chain ERC20 `symbol()` / `name()` if host presentation is unavailable; the mocked `/logo.png` image is now reserved for the neutral preview fixture only. Do not call private token metadata APIs from `Component.tsx`.
- For chain-specific write flows, use `sdk.wallet.isWrongNetwork` and the shared `sdk.wallet.switchChain()` helper or render a clear switch-network state before continuing. Do not attempt writes while the connected wallet is on the wrong chain.
- Keep contract interaction inside the Vault/token/NFT/factory boundary. Reads and writes should target `context.vaultAddress`, `context.tokenAddress`, `context.factoryAddress`, payment/quote/dividend token addresses from runtime context, token/NFT addresses derived from Vault state, or fixed contract addresses declared in `match.bindings[].externalContracts`. Do not call routers, bridges, aggregators, or unrelated app contracts from a Vault package.
- The preview shell / production host owns the token breadcrumb/header, close control, `Vault Information` frame, page width, invalid-token fallback, and any standard shared summary block already rendered on the host surface. Implement Vault-specific business UI only below that frame. Do not rebuild shell chrome or duplicate a host-provided top summary/hero block inside `Component.tsx`.
- Manifest is intentionally minimal: `artifactId`, `name`, `match`, `i18n`, optional non-oracle `endpoints`, and optional reviewed `externalFrames` only. Do not add `chainIds`, `id`, `owner`, `version`, `sdkVersion`, `actions`, `oracles`, `media`, `fallback`, `contracts`, `restrictTokenAddresses`, global `tokenAddresses`, or `caPolicy`. Chain IDs and binding targets live inside `match.bindings`; each entry must be either `{chainId, factoryAddress}`, `{chainId, vaultAddresses: [vaultAddress]}`, or `{chainId, tokenAddresses: [tokenAddress, ...]}`, with at least one binding carrying `tokenAddresses` for testing. Optional `externalContracts` also belongs inside each binding entry only.
- Avoid external endpoints and external resources by default. If a special non-oracle endpoint is unavoidable, it may be predeclared in `manifest.json` as a single absolute HTTPS URL string without username/password credentials or an array of those strings so `vault:check` can quickly validate it and print URL/query details for Workbench review. Direct `fetch(...)` targets must be static absolute HTTPS URLs covered by that declaration. Declaration does not guarantee approval. Oracle usage must use `sdk.readOracle(...)`; unprovisioned oracle IDs are blocking, provisioned oracle IDs are printed with endpoint and parameter details in Workbench review output, and oracle config is not declared in manifest.
- Avoid external frames by default. If a display-only market chart is unavoidable, declare at most one entry in `manifest.externalFrames[]` with `id`, `provider`, `src`, and `title`; supported providers are only `tradingview`, `dexscreener`, and `coingecko-terminal` (GeckoTerminal domain). `src` must be a complete static HTTPS provider URL with a fixed non-empty query string, no credentials, and no hash. Component code may render at most one `ReviewedFrame` from `@/src/ui`, using static string literal `frameId`, `provider`, `src`, and `title` props that exactly match the manifest declaration. Do not compose iframe URLs dynamically, use `srcDoc`, or use frames for quotes, risk, settlement, wallet connection, transactions, or postMessage data.
- Do not add arbitrary external navigation. Component-owned links should stay on the active chain explorer via `context.explorerBaseUrl` / `AddressLink`. Other destinations are treated as phishing-sensitive and blocked unless Flap review explicitly allowlists the URL prefix for metadata/oracle inspection.
- The only allowed local relative import is `./VaultABI`. Do not use `./helpers`, `../VaultABI`, nested component imports, local asset imports, or dynamic imports.
- The template package is a source package for the Flap Artifact Workbench. Runtime Blob/R2/S3 artifacts must be Flap-built readable ESM JS (`component.mjs`), not TSX source, not hand-uploaded JS, and not minified by default.
- Keep ABI fragments minimal.
- If `VaultABI.ts` uses human-readable ABI strings such as `"function foo() view returns (...)"`, wrap them with `parseAbi([...])` from `viem`; raw signature string arrays are not valid runtime ABI objects and `vault:check` blocks them.
- `sdk.readContract` returns ABI methods with multiple return values as tuple arrays, even when the outputs are named in a human-readable ABI. Type those reads as tuples such as `readonly [currentPool: bigint, totalReceived: bigint]`, then map tuple indexes into object-shaped UI state. Do not type multi-output reads as object interfaces. A single returned Solidity `tuple` / struct output may still be read as an object when the ABI declares it as one output with `components`.
- Keep all user-facing copy in `i18n.json`.
- Do not use direct wallet APIs, raw iframe, eval, script injection including `document.write` / `document.writeln`, unsafe inline SVG JSX, runtime remote imports, CommonJS `require(...)`, symlinks, browser storage/navigation/worker/cross-context/permission APIs, direct browser network/media APIs, undeclared endpoints/resources/frames, or undeclared fixed contract targets.
- Do not add third-party images; media is controlled by Flap runtime/Artifact Workbench policy.
- Register every Vault folder name in `src/vaults/index.ts` so `/{folder-name}` preview works.
- Registration is local preview wiring only, not production publish registration. `yarn vault:scaffold <folder-name>` does it automatically; if the four Vault files already exist, run `yarn vault:register <folder-name>` instead of editing `src/vaults/index.ts` by hand.
- All Vault CLI failures must be treated as machine-readable JSON. Read `code`, `fixHint`, and `agent.nextActions`; fix those items before retrying.
- Tool-specific entry points are compatibility wrappers only: `CLAUDE.md` for Claude Code, `GEMINI.md` for Gemini CLI, `.windsurfrules` for Windsurf, `.github/copilot-instructions.md` for GitHub Copilot, `.cursor/rules/flap-vault-ui.mdc` for Cursor, and `.cursorrules` for legacy Cursor. Keep canonical rules here, in `agent-contract.json`, `docs/ai-agent.md`, `docs/agent-entrypoints.md`, `docs/ui-pattern-snippets.md`, and `skills/flap-vault-ui-generator/SKILL.md`.
- `yarn vault:check`, `yarn build`, `yarn runtime:package`, and `yarn vault:package` verify the local `package.json` version and git history against npm latest `@flapsdk/vault-runtime`; a stale local version must fail with `template-freshness/npm-outdated`, and a version-only edit without the npm latest source commit must fail with `template-freshness/npm-git-head-mismatch`.
- Run `yarn vault:check <folder-name>` before packaging. The JSON output includes `agent.nextActions`; fix all blocking issues before finishing.
- Run `yarn vault:e2e <folder-name>` before packaging. It must cover PC / iPad / H5, `default` / `internal-market` / `dex-listed`, wrong-network UI, and layout blockers through deterministic Playwright DOM/layout/state checks, not AI image judgment. The report at `dist/e2e/<folder-name>/qa-report.json` must bind to current source hashes and use a test token declared in manifest `match.bindings[].tokenAddresses`; local `--token 0x...` overrides are only for developer self-test and do not satisfy `vault:check`. On first local runs, especially Windows, install Chromium with `yarn playwright install chromium` if `vault:e2e` returns `vault-e2e/playwright-browser-missing`.
- After `yarn vault:package <folder-name>`, run `yarn vault:verify-package dist/<folder-name>.zip` to validate the source zip marker, E2E report, file list, metadata, and hashes from the Workbench side.
- Source packages use format version `4` and include `qa/e2e-report.json` plus an `e2e` summary in `flap-vault-package.json` and `package-metadata.json`; do not hand-edit or omit this proof.
- Do not treat a developer-local tx hash or wallet trace as strong proof that a future write transaction was initiated by the local UI. Strong write-UI origin assurance requires a platform-controlled Playwright + wallet runner.
- For code-base changes, run `yarn ci` when feasible; it covers lint, typecheck, checker selftest, full three-viewport example E2E/package/verify, build, runtime package checks, and preview smoke.
